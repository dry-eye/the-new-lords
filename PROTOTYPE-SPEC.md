# New Lords — Prototype Spec & Design Coverage

Специфікація веб-прототипу, зібраного на основі `DESIGN.md`, і чесний аудит: **що з дизайну реалізовано, що частково, чого немає.**

- **Артефакт:** `new-lords-prototype.html` — один самодостатній файл (Three.js r128 з CDN, без збірки, без залежностей).
- **Джерело істини по дизайну:** `DESIGN.md` (1597 рядків). Цей документ **не замінює** його, а вимірює прототип відносно нього.
- **Мета прототипу:** перевірити **системи** (організації, економіка, торгівля, спостереження/втручання) до того, як писати це в Unreal. Графіка — рівно настільки, щоб системи були читабельні.
- **Статус на цю дату:** ядро світу + економіка + торгівля + оргполітика працюють; бій — заглушка; тактичний і транспортний шари не починались.

## 2. Що прототип уже доводить

Ці твердження перевірені headless-прогонами (600–800 тіків, 3 seed'и), а не «на око»:

1. **Замкнена економіка тримається.** Усі 6 ресурсів виробляються і споживаються; жоден не є мертвим кінцем. Населення росте на 2 з 3 seed'ів; на третьому світ біднішає — і це не баг, а економіка під тиском (дизайн саме цього й хоче).
2. **Каравани реально закривають дефіцити.** ~400–1600 рейсів за прогін, торгові доми виходять у плюс і самі купують нові каравани, потім купують перк «Логістичний хаб» на розширення місткості. Тобто **торгівля — це самопідсилювальний контур**, а не декор.
3. **Utility-AI орга дає різну поведінку від рис лідера.** Амбітні орги будують, наймають, лізуть у незалежність; пасивні сидять тихо.
4. **Крихкість ланцюга — реальна.** Перший баланс дав петлю смерті: нема інструментів → усі підприємства стоять → нема інструментів. Це **знахідка дизайну, а не інженерії**: інструменти не можуть бути жорстким гейтом, тільки множником ефективності. Записано в «Відхилення» нижче.
5. **Кількість оргів вибухає.** З 100 стартових до 300–400 за 800 тіків (кожне підприємство → бізнес-орг, кожне захоплення → політичний орг). Дизайн це передбачає (LOD для оргів), але **бюджет оргів треба закладати з першого дня**, не потім.

## 3. Архітектура прототипу

Один файл, чотири шари. Симуляція **повністю відокремлена** від рендеру — її можна вирізати й запустити в Node (я так і тестував баланс).

```
┌ SIM (детермінований, seed → світ) ─────────────────────────┐
│  noise/fBm → elevation → clim ate → zones → worldgen        │
│  economyStep()  → виробництво, споживання, ціни, податок    │
│  caravanStep()  → арбітраж, маршрути, зростання торгдомів   │
│  orgStep()      → utility-AI, слоти, зарплати, спадкоємство │
│  squadStep()    → рух, мінімальний бій, захоплення          │
└─────────────────────────────────────────────────────────────┘
┌ RENDER ─────────────────────────────────────────────────────┐
│  планета: запечена кліматична текстура + фрагментний шейдер  │
│  іконки: screen-space спрайти (стала читабельність)         │
│  близький LOD: процедурне місто (квартали → забудова)       │
└─────────────────────────────────────────────────────────────┘
┌ UI ──────────────────────────────────────────────────────────┐
│  інспектор (контент = функція від типу вибраного об'єкта)    │
│  граф організацій (кластери за фракціями, covert-фільтр)     │
│  ринок / лог подій / контроль часу                           │
└──────────────────────────────────────────────────────────────┘
```

**Тік:** 420 мс ігрового часу; швидкості 1× / 3× / 8×. Рендер інтерполює позиції юнітів між тіками (`slerp(prevDir, dir, tickAlpha)`), тому рух неперервний на будь-якій швидкості.

**Продуктивність:** 10–17 мс/тік при ~350 оргах, 26 поселеннях, ~50 загонах, ~40 караванах. Генерація світу ~1 с (переважно запікання текстури 1536×768).

## 5. Свідомі відхилення від дизайну ⚠️

Кожне — з причиною. Це матеріал для рішень, а не борг за замовчуванням.

1. **Інструменти — множник ефективності, а не жорсткий гейт.**
   Дизайн має «кожне підприємство споживає інструменти». Буквальна реалізація (нема інструментів → виробництво = 0) дала **петлю смерті**: інструменти закінчуються → майстерня теж стоїть → інструментів більше ніколи не буде. Прототип робить `efficiency = 0.55 + 0.45·(є інструменти)`. Економіка просідає, але може вибратись.
   → **Рішення для дизайну:** або так, або треба окремий «bootstrap»-механізм (ручна праця без інструментів).

2. **Масштаб підприємства залежить від тиру поселення** (`camp 0.7× … megalopolis 4.5×`).
   Без цього ферма в мегаполісі виробляє стільки ж, скільки в таборі, і мегаполіс на 14 000 душ гарантовано голодує. Дизайн цього множника не має.
   → **Рішення для дизайну:** або множник, або кілька окремих ферм на місто (тоді потрібен worker pool).

3. **Terrain zones — кліматичне поле, а не тесселяція.**
   Дизайн описує ~12 seed-точок і тесселяцію. Це дає **квадратні кордони біомів** (перевірено — виглядає погано). Прототип: біом = функція від температури (широта − висота) × вологості (свій fBm) × висоти. Кордони органічні, кольори змішуються неперервно.
   → **Рекомендація:** замінити в DESIGN.md. Тесселяція не дає нічого, чого не дає клімат.

4. **Місто генерується від кварталів, а не від вулиць.**
   Дизайн (і phiresky/procedural-cities) описують ріст вуличних сегментів. У прототипі це дало спагеті. Прототип ріже диск міста рекурсивно на квартали; **вулиця — це коридор між кварталами**, тому вона за побудовою пряма, тонка і зв'язана. Квартал забудовується по периметру з двором усередині.
   → **Рекомендація:** зафіксувати як метод для Unreal.

5. **Павн — не сфера з двома руками.**
   Дизайн задає «сфера-тіло + дві сфери-руки». На глобусі це читається як випадкові кульки. Прототип: **іконка людини** на середньому зумі, **капсула з котячою головою** на близькому. Триасферний примітив лишається валідним для тактичного шару (Scale B), де він потрібен для перевірки масштабу.

6. **Squad-орги не відколюються.**
   Дизайн дає лояльність і сецесію всім підлеглим оргам. Для загонів це давало абсурд («загін проголосив незалежність»). Прототип: загони при несплаті **дезертирують** (втрачають силу), як і описано в squad upkeep.

---

## 6. Що дизайн просить, а прототип показує, що це болітиме

- **Бюджет організацій.** 100 → 350+ за 800 тіків, і жоден не помирає. Дизайн має org-dormancy, але **org lifecycle (dissolution / merger) не описаний** — а без нього число оргів монотонно росте. Це не оптимізація, це дизайн-дірка.
- **Дефіцит як норма.** Сталь/інструменти/зброя стабільно в дефіциті у ~20 із 26 поселень. Це не поламка — це попит, який годує караванів. Але **треба вирішити**, чи це бажаний стан, чи економіка має сходитись у рівновагу (відкрите питання «asymmetric ratios» в DESIGN.md — прототип голосує за «нехай тисне»).
- **Захоплення міста завжди створює новий орг.** Оскільки з трьох опцій є лише REPLACE, кожна війна = +N оргів. PUPPET і MUTATE не просто «фічі» — вони **клапани для росту числа оргів**.
- **Scale A vs Scale B.** Прототип має тільки Scale A і він детермінований. Відкрите питання дизайну («чи можна Scale B рандомізувати») набуває ваги, щойно з'явиться Scale B: якщо так — бій розв'язуватиметься по-різному залежно від зуму гравця.

---

## 7. Наступні кроки, за пріоритетом

1. **Бій (Scale B).** Павни, класи зброї, дистанції, укриття; зв'язок «зброя зі складу → сила загону». Плюс ратифікувати Scale A/B reconciliation.
2. **Org lifecycle + capacity.** Розпуск, злиття, ліміти на пряме володіння — інакше світ тоне в оргах.
3. **Перки.** Головний важіль реіграбельності за дизайном, а в прототипі їх один.
4. **Covert exposure.** Таємні ребра створюються, але ніколи не розкриваються — половина механіки.
5. **Worker pool + construction as market.** Зараз праця й будівництво — це числа, а не події на мапі.
6. **PUPPET / MUTATE при захопленні.**
