# New Lords — Pipeline розробки

Черга задач — GitHub Issues репо `dry-eye/the-new-lords`. Джерело істини дизайну — `docs/DESIGN.md`. Статус задачі = label.

## Статуси (labels)

| Label | Значення |
|---|---|
| `status:to-design` | Треба дизайн-рішення — у код **не брати** |
| `status:to-do` | Готове до коду — worker може брати |
| `status:in-progress` | Worker узяв (займає слот пулу) |

Done = PR замерджено (squash) — issue closed через `Closes #N`.

## Ролі

**Worker** — довготривала loop-сесія, тримає пул ≤10 задач у роботі. Щотіку добирає задачі зі `status:to-do` до максимуму (менше 10 — лише якщо готових задач менше), кожну веде суб-агент у власному git-worktree. Взяв → `in-progress`. Готово (build+verify зелені) → відкриває PR з `Closes #N` і **сам squash-мержить** у `main` (auto-merge, без людського рев'ю). Щось неясно → повертає issue в `to-design` з коментарем «що саме неясно».

**Design-сесія** (skill, спирається на `sync-design`) — розбирає `to-design`: уточнює/брейнштормить через інтерв'ю, фіксує рішення в `docs/DESIGN.md`, повертає issue в `to-do` (або нарізає нові issue).

**Feedback-сесія** — ти дивишся `main`, даєш фідбек; вона **одразу** нарізає кожен пункт фідбеку в окремий issue — без уточнюючих питань до тебе, без накопичення й без очікування «go». Класифікує лейблом: усе зрозуміло → `status:to-do`; є відкриті питання → `status:to-design` (перелічені в тілі issue). **Уточнення тут не робляться** — усі відкриті питання й дизайн-рішення виносяться на дизайн-сесії (`/designsession`).

## Потік

```
to-design → to-do → in-progress → PR (squash self-merge) → issue closed
    ↑___ worker повертає, якщо неясно ___|
```

Відкат: 1 задача = 1 squash-коміт → Revert на її PR відкочує рівно її.

## Конвенції

**Мова.**
- Чат із користувачем — українською.
- **GitHub — англійською**: заголовки й тіла issue, коментарі, назви й описи PR.
- Дизайн-доки (`docs/DESIGN.md`, специ на кшталт `docs/PATHFINDING_SPEC.md`) і commit-меседжі — англійською.
- Ідентифікатори коду й технічні терміни — в оригіналі.

**Дизайн-сесія — лише дизайн.** Код прототипу в дизайн-сесії не писати: фіксувати рішення в `docs/DESIGN.md`/специ та заводити issue; реалізує worker і мерджить у `main`. «Треба зробити X» = завести задачу на X. Виняток — явний наказ кодити в сесії, і то у власному worktree/чіпі, не в головній робочій теці.

**Задачі.** Не питати «go» перед записом/заведенням — діяти одразу; завжди завершувати вирішене tracked-issue. Коли сказано опрацьовувати чергу — іти по `to-design` автономно, рішення власним судженням, без питань. Закриваєш design-issue замість relabel — **зніми мітку `status:to-design`** (закрите не має висіти в черзі).

**Візуальні рішення.** Для вибору вигляду рендерити 2–3 варіанти візуально (композит) і давати обирати з побаченого — не питати текстом.

**Git.** Ізольовані коміти (зміну дизайн-доку — окремо); auto-push одразу після коміту; при відмові push — `git pull --rebase origin main` і повторити.

**Джерело істини.** `docs/DESIGN.md` (+ специ) — єдине джерело; прототип (`new-lords-prototype.html`) — тонке чесне відображення. New Lords ізольований — сусідній `M:/Projects/PowerMap` не читати.

**Верифікація прототипу.** WebGL не рендериться в in-app-превʼю (вьюпорт 0×0) — логіку перевіряти page-eval'ом (`generate()`/`buildScene` без помилок консолі), візуал — у власному браузері (`python -m http.server` → відкрити файл).
