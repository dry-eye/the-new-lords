# New Lords — Прототип: глобус, граф організацій, інспектор

Дизайн-спека для першого прототипу New Lords: 3D-глобус зі статичним, детермінованим світом, оверлей графа організацій і наскрізний інспектор із крос-навігацією між усіма сутностями. Мета — довести UX "клікни на будь-що → перестрибуй між пов'язаними сутностями одним кліком", а не побудувати живу симуляцію.

---

## 1. Мета і межі обсягу

**Що доводимо:** що граф організацій (вузли-орги, ребра-слоти/agent-зв'язки, ownership-ребра до власності) і 3D-глобус із поселеннями/підприємствами/караванами/загонами можуть співіснувати з одним інспектором, у якому кожна сутність клікабельна й веде до кожної пов'язаної сутності.

**Свідомо поза обсягом v1:**
- Сімуляційний тік — жодна механіка не тікає сама по собі (виробництво, бій, рух караванів, лояльність, entrenchment-прогрес тощо). Усе, що в DESIGN.md залежить від тіка, у прототипі показане, але задизейблене (див. §7).
- Player Mode / вибір персонажа на старті сесії — редагування організацій не гейтиться на "чи це твоя організація", бо гри-як-такої (з переможцем/програвшим) немає.
- Бойова система, паливо/боєприпаси, Squad Loadout (по-пешна розкладка місць) — транспорт спрощений до одного зв'язку "загін ⇄ транспорт", без посадкових ролей.
- LOD-тіри й інтер'єри будівель — глобус має один рівень деталізації, без зуму в будівлю.
- Term-tooltip глосарій (наведення на термін → спливаюче пояснення) — корисна ідея з DESIGN.md, але не вирішено для v1; не будується цим проходом.

**Джерело істини для ігрових механік:** [`docs/DESIGN.md`](../../DESIGN.md). Ця спека описує лише те, як прототип *показує й обмежено взаємодіє* з тим, що там визначено — вона не дублює саму механіку.

## 2. Стек і базові принципи

- **TypeScript + Vite + React.**
- **Глобус** — 3D-сцена на react-three-fiber.
- **Граф організацій** — SVG-оверлей із force-directed розкладкою.
- **Інспектор** — звичайні React-панелі, одна активна панель за раз (перемикається, не накладається).
- **Тематизація** — токен-based theme-provider (CSS custom properties/дизайн-токени), не хардкод кольорів по компонентах. Жодного світлого стилю — див. §10.
- **Локалізація** — рядки інтерфейсу через словники, не інлайн-текст у компонентах. Див. §9.

## 3. Джерело даних і масштаб світу

Детермінований seeded worldgen: статичний знімок світу з фіксованого seed, без сімуляційного тіка. Світ можна перегенерувати, ввівши новий seed (миттєва заміна всього знімка, не інкрементальна зміна).

| Параметр | Значення |
|---|---|
| Поселення (village/city/megalopolis) | ~30 |
| Терен-зони (гори/ліс/рівнина, Poisson-disc — дефолт із DESIGN.md Worldgen) | ~15–18 |
| Дороги | MST + shortcut-links між поселеннями (як у DESIGN.md) |
| Каравани | ~40, статичні маркери на дорогах |
| Загони | ~40 (гарнізони при частині поселень + кілька незалежних/бандитських) |
| Агентурні мережі | невелика кількість для демонстрації механіки (орієнтовно 2–3) — конкретне число уточнюється на етапі плану |
| Транспорт | 0 у стартовому знімку — з'являється лише через debug-панель (§7) |

Outside-type підприємства (шахта/лісопилка/ферма/нафтовежа) сидять біля прилеглих терен-зон, по одному на відповідне поселення, власник — поселенська організація. Inside-type підприємства (forge/armory/factory/component factory/construction company) розподіляються за існуючою тір-таблицею DESIGN.md (town/city/megalopolis), без окремого маркера (§4).

**Кількість організацій окремо не задається** — вона є похідною: кожне поселення це political-орг, кожне outside-підприємство — окремий business-орг його власника, кожен загін — сам собі squad-орг (за правилом DESIGN.md "every squad is also an organization"), плюс невелика кількість `network`-orgs (агентурні мережі, див. вище).

## 4. Сутності на глобусі (клікабельні маркери)

| Сутність | Що це | Джерело |
|---|---|---|
| Поселення | village / city / megalopolis | worldgen |
| Підприємство (outside-type) | шахта, лісопилка, ферма, нафтовежа — власний маркер | worldgen, прив'язане до терен-зони |
| Дорога | ребро між поселеннями | worldgen — **не клікабельна**, фон для караванів |
| Караван | статична точка на дорозі | worldgen |
| Загін | статичний маркер (гарнізон/незалежний/бандитський) | worldgen |
| Транспорт | з'являється лише через debug-панель, поки не сів у нього загін | debug-дія (§7) |

**Поза глобусом, лише в панелях:**
- Inside-type підприємства — записом у списку панелі поселення; відкривають той самий Enterprise-інспектор, що й outside-type.
- Персонажі — клікабельне ім'я всередині панелі організації/поселення; власного маркера немає (камера "телепортується" до орга-роботодавця, за конвенцією DESIGN.md).

## 5. Інспектори — параметри, функції, навігація

Спільне правило для всіх: клік по будь-якому підкресленому імені відкриває панель цієї сутності **і** центрує глобус/граф на ній. Відкрита завжди рівно одна панель.

### 5.1 Поселення (Settlement)
- **Параметри:** назва, тип, площа території, дохід (база + бонус території з розбивкою), ресурси з чистим виробництвом/крок, підсумок підприємств (працює/простоює).
- **Функції:** немає — чисто похідний показ, усі важелі належать орг-власнику.
- **Переходи:** → власник-орг, → батько власника, → кожне підприємство зі списку.

### 5.2 Підприємство (Enterprise) — єдиний тип для outside + inside
- **Параметри:** тип-префіксна назва, вид (шахта/лісопилка/ферма/нафтовежа/forge/armory/factory/component factory/construction company), штат є/немає, вихід/тік, поточний запас.
- **Функції:** немає.
- **Переходи:** → власник-орг, → поселення-господар.

### 5.3 Караван (Caravan)
- **Параметри:** маршрут (звідки → куди), тип перевізника, фаза вантажу.
- **Функції:** немає.
- **Переходи:** → поселення-звідки, → поселення-куди, → власник-орг.

### 5.4 Загін (Squad)
- **Параметри:** id, фракція, сила, режим контролю (auto/manual — статичний показ, без ефекту), статус наказу (статичний показ), ціль гарнізону, змонтований транспорт (якщо є).
- **Функції:**
  - **Сісти в транспорт** — активна, якщо на позиції загону є запаркований транспорт; миттєво монтує.
  - **Покинути транспорт** — активна, якщо загін змонтований; миттєво демонтує, транспорт лишається запаркованим на поточній позиції.
  - Move/Attack/Patrol/інші order-дії — присутні в UI, задизейблені з тултіпом (§7).
- **Переходи:** → ціль гарнізону, → змонтований транспорт, → власна організація.

### 5.5 Транспорт (Vehicle) — новий тип, з'являється лише через debug-панель
- **Параметри:** тип (авто/байк/ван/шатл/НРК/БМП/combat shuttle/heavy UGV), місткість + сімейство (повітря/земля), власник-орг, змонтований загін (якщо є) або "Запаркований".
- **Функції:** немає власних (Board/Leave керуються з панелі загону).
- **Переходи:** → власник-орг, → змонтований загін.

### 5.6 Організація (Organization) — єдина повністю редагована панель
- **Параметри:** назва, лідер, спадкоємець, кількість членів, ambition/утиліті-вага, kind (тепер включно з `network`), leadership type, tax rate, income sources, власність (список), контрольовані слоти (окупант/вид/salary), agent-едж (для `network`-kind: орг-носій + доступні дії), дочірні орги (рекурсивно), breadcrumb предків.
- **Функції:** змінити kind, змінити leadership type, змінити tax rate, перемкнути income sources, вибрати спадкоємця, створити/розірвати слот, змінити salary слота, ініціювати intel-звіт/платний саботаж через agent-едж.
- **Переходи:** → лідер, → спадкоємець, → кожен об'єкт власності, → окупант кожного слота, → орг-носій кожного agent-еджу, → дочірні орги, → предки в breadcrumb.

### 5.7 Персонаж (Character)
- **Параметри:** ім'я, вік/категорія, риси характеру, поселення проживання, основне членство в організації, вторинне членство в агентурній мережі (якщо завербований), орг, яким керує (+ CK-титул), спадкоємець.
- **Функції:** немає — спадкоємець редагується з панелі організації.
- **Переходи:** → поселення проживання, → орг основного членства, → орг вторинного (мережевого) членства, → орг, яким керує, → спадкоємець.

## 6. Граф організацій

**Вузли:** організації (kind: political/business/squad/network; leadershipType) + вузли власності (поселення/підприємства, прив'язані ownership-ребром).

**Ребра:**
| Вид | Стиль | Видимість |
|---|---|---|
| control | суцільна лінія | завжди публічна |
| influence / investment | пунктир | завжди публічна |
| alliance | подвійна лінія | завжди публічна |
| covert | (окремий стиль) | прихована — лише власнику або якщо викрита |
| agent (мережа → орг-носій агента) | пунктирна крапка (dotted) | прихована — та сама умова, що й covert |
| ownership (орг → власність) | окремий стиль, відмінний від 5 видів ребер вище | завжди публічна |

**View-тумблери:**
- **Hide property** — ховає вузли власності й ownership-ребра, лишає тільки орг-до-орг.
- **Focus on selection** — при вибраній організації показує лише досяжні через будь-яке ребро (враховуючи повний граф, незалежно від того, чи намальовані covert/agent наразі).
- **Viewpoint** — один контрол, за замовчуванням **Спектатор** (всевидячий: усі ребра, включно з covert/agent, незалежно від власності/викриття). Перемикання на конкретну організацію X застосовує фільтр учасника: публічні ребра — завжди видно; приховані (covert/agent/secret-ownership) — лише якщо X власник або вже викрито проти X.

## 7. Взаємодія: транспорт, накази, drag, debug-панель

**Board/Leave** — реально функціональні, миттєві (без сімуляції — це чиста зміна стану):
- Сісти: потрібен запаркований транспорт на позиції загону.
- Покинути: транспорт лишається запаркованим маркером там, де стояв загін.
- Візуально: у змонтованому стані загін+транспорт рендеряться одним об'єднаним маркером (окремий маркер транспорту ховається); після Leave — знову два окремі маркери.

**Симуляційно-залежні дії лишаються видимими, але задизейбленими.** Move, Attack, Patrol, Garrison-as-order, Capture-on-arrival та будь-яка інша order-дія з ABILITIES.md, а також будь-яка тік-залежна механіка DESIGN.md (Assault, entrenchment-прогрес, авто-хоп караванів, Wander/Reconnect) — показані в UI (кнопка/пункт меню), неактивні, з тултіпом **"Потребує симуляційного тіка — заплановано на майбутню версію."**

**Drag-переміщення** — стенд-ін замість відсутньої симуляції руху: будь-яку позиційну сутність на глобусі (поселення, підприємство, загін, транспорт, караван) можна перетягнути мишею в нову точку — миттєва зміна позиції, без шляху/доріг/анімації подорожі.

**Debug-панель** — окрема від гравцівського інспектора панель для розробницьких дій, щоб не засмічувати інспектор кнопками, яких у грі не буде. Доступна власним перемикачем/вкладкою в UI (не змішана з гравцівськими панелями інспектора):
- **"Додати обраному загону: [тип транспорту ▾]"** — миттєво створює обраний тип транспорту (повний перелік DESIGN.md) і одразу монтує на виділений загін.

## 8. Агентурні мережі — залежність від DESIGN.md

Ігрова механіка (новий `kind: 'network'`, виняток мульти-членства, agent-едж, дві дії каналом — intel-звіт і платний саботаж) уже описана в [`docs/DESIGN.md`](../../DESIGN.md) (секція "Agent networks — the `network` kind" і суміжні). Прототип лише відображає й дозволяє редагувати цю модель через Organization-інспектор і Organization Graph (§5.6, §6) — не додає нових ігрових правил понад те, що вже в DESIGN.md.

Так само вже внесені в DESIGN.md: Enterprise- та Vehicle-інспектори (як рядки в "Inspector — object types & fields"), поле "mounted vehicle" у Squad, Viewpoint-тумблер у "Organization Graph — property nodes and view filters".

## 9. Локалізація

Українська (за замовчуванням) + англійська, з перемикачем мови в UI. Весь текст інтерфейсу винесений у словники, не захардкоджений у компонентах.

---

## 10. Тематизація (стилі)

Токен-based theme-provider: набори дизайн-токенів (кольори, типографіка, відступи), а не хардкод по компонентах. Жодного світлого стилю. Два початкові стилі для перевірки розширюваності архітектури:
- **Мінімалізм** — темний, малоконтрастний, мало прикрас.
- **Бруталізм** — темний, високий контраст, "оголена" сітка/типографіка.

Перемикач стилів у UI — миттєва заміна активного набору токенів. Додавання нового стилю в майбутньому не повинно вимагати змін у компонентах, лише новий набір токенів.

## 11. Відкриті питання

- Точна кількість агентурних мереж і персонажів-агентів у стартовому знімку (§3) — орієнтовне число, уточнюється на етапі плану.
- Term-tooltip глосарій — свідомо не вирішено для v1 (див. §1); можливе розширення пізніше.
- Розподіл поселень за тірами (village/city/megalopolis) серед ~30 — не зафіксовано конкретними числами, лишається деталлю етапу плану.
