# New Lords — Макет: глобус, граф організацій, інспектор

Дизайн-спека для першого макета New Lords (наступна стадія після макета — прототип, окрема спека): 3D-глобус зі статичним, детермінованим світом, оверлей графа організацій і наскрізний інспектор із крос-навігацією між усіма сутностями. Мета — довести UX "клікни на будь-що → перестрибуй між пов'язаними сутностями одним кліком", а не побудувати живу симуляцію.

---

## 1. Мета і межі обсягу

**Що доводимо:** що граф організацій (вузли-орги, ребра-слоти/agent-зв'язки, ownership-ребра до власності) і 3D-глобус із поселеннями/підприємствами/караванами/загонами можуть співіснувати з одним інспектором, у якому кожна сутність клікабельна й веде до кожної пов'язаної сутності.

**Свідомо поза обсягом v1:**
- Гейплейний тік (виробництво, бій, рух караванів, лояльність, entrenchment-прогрес тощо) — усе, що DESIGN.md рухає в просторі/часі, у макеті показане, але задизейблене (§7). Окремий, легкий фоновий тік (§13) все ж є — він живить LOD, org-рішення й перки, не гейплей.
- Player Mode / вибір персонажа на старті сесії — редагування організацій не гейтиться на "чи це твоя організація", бо гри-як-такої (з переможцем/програвшим) немає.
- Бойова система, паливо/боєприпаси, Squad Loadout (по-пешна розкладка місць) — транспорт спрощений до одного зв'язку "загін ⇄ транспорт", без посадкових ролей.
- Інтер'єри будівель (заходити всередину, кімната за кімнатою) — будівлі на L2 суто зовнішні, косметичні, без власного інспектора. LOD-тіри рендеру/sim-fidelity тепер **у обсязі** — див. §12.
- Term-tooltip глосарій (наведення на термін → спливаюче пояснення) — корисна ідея з DESIGN.md, але не вирішено для v1; не будується цим проходом.
- **Ієрархічний зв'язок і прихованість — обидва вирішено й уже в DESIGN.md.** Capacity/founding-механіка (Split/Recall, "Заснувати підпорядкованого" — §5.4, §5.6) і уніфікована visibility-властивість (`public`/`covert` на hierarchy/influence/agent, замінює й covert-як-kind, і secret ownership як окрему механіку — §6). Нічого з цього більше не відкладено.

**Джерело істини для ігрових механік:** [`docs/DESIGN.md`](../../DESIGN.md). Ця спека описує лише те, як макет *показує й обмежено взаємодіє* з тим, що там визначено — вона не дублює саму механіку.

## 2. Стек і базові принципи

- **TypeScript + Vite + React.**
- **Глобус** — 3D-сцена на react-three-fiber.
- **Граф організацій** — SVG-оверлей із force-directed розкладкою.
- **Інспектор** — звичайні React-панелі, одна активна панель за раз (перемикається, не накладається).
- **Тематизація** — токен-based theme-provider (CSS custom properties/дизайн-токени), не хардкод кольорів по компонентах. Жодного світлого стилю — див. §10.
- **Локалізація** — рядки інтерфейсу через словники, не інлайн-текст у компонентах. Див. §9.

## 3. Джерело даних і масштаб світу

Детермінований seeded worldgen: статичний знімок світу з фіксованого seed, без гейплейного тіка (легкий фоновий тік — §13 — усе ж є). Світ можна перегенерувати, ввівши новий seed (миттєва заміна всього знімка, не інкрементальна зміна).

| Параметр | Значення |
|---|---|
| Поселення (village/city/megalopolis) | ~30 |
| Терен-зони (гори/ліс/рівнина, Poisson-disc — дефолт із DESIGN.md Worldgen) | ~15–18 |
| Дороги | MST + shortcut-links між поселеннями (як у DESIGN.md) |
| Каравани | ~40, статичні маркери на дорогах |
| Загони | ~40 (гарнізони при частині поселень + кілька незалежних/бандитських) |
| Агентурні мережі | невелика кількість для демонстрації механіки (орієнтовно 2–3) — конкретне число уточнюється на етапі плану |
| Транспорт | 0 у стартовому знімку — з'являється лише через debug-панель (§7) |

Outside-type підприємства (шахта/лісопилка/ферма/нафтовежа) сидять біля прилеглих терен-зон, по одному на відповідне поселення, власник — поселенська організація. Inside-type підприємства (forge/armory/factory/component factory/construction company) розподіляються за існуючою тір-таблицею DESIGN.md (town/city/megalopolis), без окремого маркера (§4).

**Кількість організацій окремо не задається** — вона є похідною: кожне поселення це political-орг, кожне outside-підприємство — окремий business-орг його власника, кожен загін — сам собі squad-орг (за правилом DESIGN.md "every squad is also an organization"), плюс невелика кількість `network`-orgs (агентурні мережі, див. вище).

## 4. Сутності на глобусі (клікабельні маркери)

| Сутність | Що це | Джерело |
|---|---|---|
| Поселення | village / city / megalopolis | worldgen |
| Підприємство (outside-type) | шахта, лісопилка, ферма, нафтовежа — власний маркер | worldgen, прив'язане до терен-зони |
| Дорога | ребро між поселеннями | worldgen — **не клікабельна**, фон для караванів |
| Караван | статична точка на дорозі | worldgen |
| Загін | статичний маркер (гарнізон/незалежний/бандитський) | worldgen |
| Транспорт | з'являється лише через debug-панель, поки не сів у нього загін | debug-дія (§7) |

**Поза глобусом, лише в панелях:**
- Inside-type підприємства — записом у списку панелі поселення; відкривають той самий Enterprise-інспектор, що й outside-type.
- Персонажі — клікабельне ім'я всередині панелі організації/поселення; власного маркера немає (камера "телепортується" до орга-роботодавця, за конвенцією DESIGN.md).

## 5. Інспектори — параметри, функції, навігація

Спільне правило для всіх: клік по будь-якому підкресленому імені відкриває панель цієї сутності **і** центрує глобус/граф на ній. Відкрита завжди рівно одна панель.

### 5.1 Поселення (Settlement)
- **Параметри:** назва, тип, площа території, дохід (база + бонус території з розбивкою), ресурси з чистим виробництвом/крок, підсумок підприємств (працює/простоює).
- **Функції:** немає — чисто похідний показ, усі важелі належать орг-власнику.
- **Переходи:** → власник-орг, → батько власника, → кожне підприємство зі списку.

### 5.2 Підприємство (Enterprise) — єдиний тип для outside + inside
- **Параметри:** тип-префіксна назва, вид (шахта/лісопилка/ферма/нафтовежа/forge/armory/factory/component factory/construction company), штат є/немає, вихід/тік, поточний запас.
- **Функції:** немає.
- **Переходи:** → власник-орг, → поселення-господар.

### 5.3 Караван (Caravan)
- **Параметри:** маршрут (звідки → куди), тип перевізника, фаза вантажу.
- **Функції:** немає.
- **Переходи:** → поселення-звідки, → поселення-куди, → власник-орг.

### 5.4 Загін (Squad)
- **Параметри:** id, фракція, сила, режим контролю (auto/manual — статичний показ, без ефекту), статус наказу (статичний показ), ціль гарнізону, змонтований транспорт (якщо є), підпорядкований загін-детач (якщо є, від Split).
- **Функції:**
  - **Сісти в транспорт** — активна, якщо на позиції загону є запаркований транспорт; миттєво монтує.
  - **Покинути транспорт** — активна, якщо загін змонтований; миттєво демонтує, транспорт лишається запаркованим на поточній позиції.
  - **Split** — активна, якщо є вільний subordinate-squad слот (база 1, DESIGN.md); миттєво відколює частину загону в новий, прямо підпорядкований загін-орг на тій самій позиції.
  - **Recall** — активна, якщо поруч є прямий підпорядкований загін; миттєво розпускає його, зливає пешки назад, звільняє слот.
  - Move/Attack/Patrol/інші order-дії — присутні в UI, задизейблені з тултіпом (§7).
- **Переходи:** → ціль гарнізону, → змонтований транспорт, → підпорядкований загін-детач (якщо є), → власна організація.

### 5.5 Транспорт (Vehicle) — новий тип, з'являється лише через debug-панель
- **Параметри:** тип (авто/байк/ван/шатл/НРК/БМП/combat shuttle/heavy UGV), місткість + сімейство (повітря/земля), власник-орг, змонтований загін (якщо є) або "Запаркований".
- **Функції:** немає власних (Board/Leave керуються з панелі загону).
- **Переходи:** → власник-орг, → змонтований загін.

### 5.6 Організація (Organization) — 4 таби: Overview / Policy / Perks / Decisions

Одна панель, чотири вкладки. Overview — чисто читання; зміни стану відбуваються лише в Policy/Perks/Decisions.

**Overview (read-only):**
- Параметри: назва, лідер, спадкоємець, кількість членів, ambition/утиліті-вага, kind (political/business/squad/network/military/intelligence/secret_police), leadership type, capacity-лічильники з таблиці DESIGN.md як "зайнято/максимум" (control-slot, subordinate-squad, subordinate-business, subordinate-political, alliance, enterprise-ownership, agent-network), власність (список), контрольовані слоти (окупант/вид/salary), agent-едж (для network/intelligence/secret_police: орг-носій + доступні дії), дочірні орги (рекурсивно, з позначкою типу зв'язку — hire-слот чи прямий підпорядкований), breadcrumb предків.
- Переходи: → лідер, → спадкоємець, → кожен об'єкт власності, → окупант кожного слота, → орг-носій кожного agent-еджу, → дочірні орги, → предки в breadcrumb.

**Policy:**
- Редаговані поля: tax rate, structuralPolicy (лише political), income sources (мультитумблер), supply policy (per-owned-settlement). Кожне поле має hover-тултіп із поясненням ефекту.
- Зміни застосовуються миттєво; похідні значення (напр. money income) перераховуються реактивно за формулою DESIGN.md — без сімуляційного тіка, суто перерахунок при зміні вхідних даних.
- Доступність полів залежить від kind (напр. structuralPolicy лише в political; income sources — підмножина, дозволена для kind, за таблицею DESIGN.md); недоступні поля приховані, а не задизейблені.

**Perks:**
- Реально функціонує (не задизейблено) — керується легким фоновим тіком (§13), не гейплейним.
- 3-card offer, коли доступний (milestone за DESIGN.md), або повний браузабельний список перків із позначкою доступності (сіра/активна за kind + leadershipType + prereq + exclusion).
- Взяття перка миттєво застосовує ефект — оновлює відповідне число на Overview (напр. subordinate-squad capacity +1). За правилом DESIGN.md "перки — це ефект, не колекція": таб **не показує список уже взятих перків** — тільки поточні цифри на Overview і запис у History.

**Decisions:**
- Змінити kind (dropdown), змінити leadership type (dropdown) — перенесено сюди з Overview.
- **Заснувати підпорядкованого** — обрати kind дитини (political/business/squad), гейт по відповідній subordinate-X capacity, дебет вартості з власної скарбниці.
- **BecomePolitical**, **Expand** — реально функціональні (керуються тим самим легким фоновим тіком, §13), не задизейблені.

**History** (доступна з будь-якого табу): хронологічний лог подій цього орга. Єдине місце, де назва взятого перка взагалі відображається (напр. "Взято перк: Free Company Charter — [дата]").

### 5.7 Персонаж (Character)
- **Параметри:** ім'я, вік/категорія, риси характеру, поселення проживання, основне членство в організації, вторинне членство в агентурній мережі (якщо завербований), орг, яким керує (+ CK-титул), спадкоємець.
- **Функції:** немає — спадкоємець редагується з панелі організації.
- **Переходи:** → поселення проживання, → орг основного членства, → орг вторинного (мережевого) членства, → орг, яким керує, → спадкоємець.

## 6. Граф організацій

**Вузли:** організації (kind: political/business/squad/network/military/intelligence/secret_police; leadershipType) + вузли власності (поселення/підприємства, прив'язані ownership-ребром).

**Ребра:** видимість тепер — **властивість** (`visibility: public | covert`) кожного ребра, не окремий вид. `covert`-прапорець можна ставити на hierarchy й influence; control/alliance завжди публічні; agent — завжди covert за замовчуванням. Те, що раніше звалось "covert-слот" чи "secret ownership" — це просто influence/hierarchy-ребро з `visibility: covert` (DESIGN.md, "Visibility — a property, not a kind").

| Вид | Стиль | `visibility` за замовчуванням |
|---|---|---|
| hierarchy (parent/child) | тонка суцільна лінія | public (covert = "secret ownership") |
| control | товща суцільна лінія | завжди public |
| influence / investment | пунктир | public (covert = "прихований слот") |
| alliance | подвійна лінія | завжди public |
| agent (мережа → орг-носій агента) | пунктирна крапка (dotted) | завжди covert |
| ownership (орг → власність) | окремий стиль, відмінний від 5 видів ребер вище | завжди public |

**View-тумблери:**
- **Hide property** — ховає вузли власності й ownership-ребра, лишає тільки орг-до-орг.
- **Focus on selection** — при вибраній організації показує лише досяжні через будь-яке ребро (враховуючи повний граф, незалежно від того, чи намальовані covert-flagged ребра наразі).
- **Viewpoint** — один контрол, за замовчуванням **Спектатор** (всевидячий: усі ребра, включно з `covert`-flagged, незалежно від власності/викриття). Перемикання на конкретну організацію X застосовує фільтр учасника: `public` ребра — завжди видно; `covert`-flagged (hierarchy, influence чи agent) — лише якщо X власник або вже викрито проти X.

## 7. Взаємодія: транспорт, накази, drag, debug-панель

**Board/Leave** — реально функціональні, миттєві (без сімуляції — це чиста зміна стану):
- Сісти: потрібен запаркований транспорт на позиції загону.
- Покинути: транспорт лишається запаркованим маркером там, де стояв загін.
- Візуально: у змонтованому стані загін+транспорт рендеряться одним об'єднаним маркером (окремий маркер транспорту ховається); після Leave — знову два окремі маркери.

**Рух/бій-залежні дії лишаються видимими, але задизейбленими.** Move, Attack, Patrol, Garrison-as-order, Capture-on-arrival та будь-яка інша order-дія з ABILITIES.md, а також будь-яка механіка DESIGN.md, що рухає сутність або накопичує ефект у просторі/часі (Assault, entrenchment-прогрес, авто-хоп караванів, Wander/Reconnect) — показані в UI (кнопка/пункт меню), неактивні, з тултіпом **"Потребує симуляційного тіка — заплановано на майбутню версію."**

**Це не поширюється на org-рішення й перки.** BecomePolitical, Expand, TakePerk (Perks-таб) і Заснувати підпорядкованого (Decisions-таб, §5.6) — реально функціональні, керовані легким фоновим тіком (§13), а не задизейблені: жодне з них не рухає сутність у просторі й не потребує тривалого бою/виробництва.

**Drag-переміщення** — стенд-ін замість відсутньої симуляції руху: будь-яку позиційну сутність на глобусі (поселення, підприємство, загін, транспорт, караван) можна перетягнути мишею в нову точку — миттєва зміна позиції, без шляху/доріг/анімації подорожі.

**Debug-панель** — окрема від гравцівського інспектора панель для розробницьких дій, щоб не засмічувати інспектор кнопками, яких у грі не буде. Доступна власним перемикачем/вкладкою в UI (не змішана з гравцівськими панелями інспектора):
- **"Додати обраному загону: [тип транспорту ▾]"** — миттєво створює обраний тип транспорту (повний перелік DESIGN.md) і одразу монтує на виділений загін.
- **Render-LOD і Sim-fidelity оверлеї** — див. §12.

## 8. Агентурні мережі — залежність від DESIGN.md

Ігрова механіка (новий `kind: 'network'`, виняток мульти-членства, agent-едж, дві дії каналом — intel-звіт і платний саботаж) уже описана в [`docs/DESIGN.md`](../../DESIGN.md) (секція "Agent networks — the `network` kind" і суміжні). Макет лише відображає й дозволяє редагувати цю модель через Organization-інспектор і Organization Graph (§5.6, §6) — не додає нових ігрових правил понад те, що вже в DESIGN.md.

## 9. Локалізація

Токен-based theme-provider: набори дизайн-токенів (кольори, типографіка, відступи), а не хардкод по компонентах. Жодного світлого стилю. Два початкові стилі для перевірки розширюваності архітектури:

## 12. LOD: рендер-тіри та sim-fidelity

DESIGN.md розділяє дві незалежні осі — **рендер-LOD** (L0/L1/L2, керується зумом камери) і **fidelity-режим симуляції** (data-only/aggregate/hot, керується ігровими сигналами — бій, наказ, подія — і явно **не** прив'язаний до камери). Макет відтворює обидві осі окремо, з debug-оверлеєм, що показує кожну явно.

### Рендер-LOD (камера-залежний)

Повний набір тірів DESIGN.md, безперервно перераховується в реальному часі під час руху камери (не одноразовий знімок), із плавним crossfade без "поп"-ефекту:

| Тір | Що видно |
|---|---|
| **L0 — Strategic** | лише кордони territorial-зон і глайфи (поселення/загони як прості крапки) |
| **L1 — Operational** | контур поселення, дороги, загони як одиничні маркери |
| **L2 — Detailed** | реальні будівлі (див. нижче), загони як окремі позначки одиниць |

**Будівлі на L2** — справжня процедурна генерація footprint-to-plan (та сама техніка, що й у DESIGN.md Worldgen), а не спрощений заглушка-кластер. Масштаб реалістичний і узгоджений між транспортом/людиною/будівлею/містом. Кількість будівель зменшена помірно (орієнтовно вдвічі, не втричі) проти "звичайного" міста — у містах усе ще багато будівель, у селищах менше. Будівлі суто косметичні: не мають власного інспектора, не клікабельні, не відкривають інтер'єр.

### Sim-fidelity (не камера-залежний)

Оскільки геймплейного тіка немає, ця вісь керується тим самим **легким фоновим тіком** (§13; тут — його LOD-драйвер-роллю), а не окремим механізмом — постійно працює в реальному часі, але не сімулює економіку/бій/лояльність, лише вирішує рівень деталізації:

- **Камера-близькість** — базовий сигнал: що ближче до камери, то вища fidelity.
- **Виділена сутність = проксі "присутності гравця"** — оскільки в макеті немає реальних наказів/бою (сигналів DESIGN.md), поточно вибрана/відкрита в інспекторі сутність трактується як "гравець тут" — її регіон лишається **hot** незалежно від того, куди зараз дивиться камера. Це пряме відтворення правила DESIGN.md "ніколи просто за тим, куди дивиться камера гравця".
- Grace-період перед демоцією (як у DESIGN.md) — коротка затримка, перш ніж hot-регіон повертається в aggregate/data-only, щоб уникнути мерехтіння на межі.

| Fidelity | Коли |
|---|---|
| **Hot (full)** | камера близько АБО ця сутність зараз виділена/відкрита в інспекторі |
| **Aggregate (cohort)** | не hot, але в межах середньої дистанції — агрегований показ |
| **Data-only** | далеко, не виділено — чисто дані, нічого не рендериться в деталях |

### Конкретні кейси

| Кейс | Рендер-LOD | Sim-fidelity |
|---|---|---|
| Камера близько до поселення | L2 (будівлі) | Hot |
| Камера далеко, нічого там не вибрано | L0 або L1 (залежно від дистанції) | Data-only / Aggregate |
| Камера далеко, але саме там вибрана/відкрита сутність | L0/L1 (візуально далеко) | **Hot** (попри дистанцію камери — див. вище) |

### Debug-оверлей (§7, debug-панель)

Два окремі, перемикні live-панелі, що явно розділяють дві осі:
- **Render-LOD оверлей** — бейдж тіру (L0/L1/L2) біля кожного видимого поселення/регіону + підсумковий лічильник "L0: X · L1: Y · L2: Z".
- **Sim-fidelity панель** — підсумковий лічильник "Hot: X · Aggregate: Y · Data-only: Z", оновлюється з того самого LOD-драйвер-циклу.

## 13. Легкий фоновий тік — спільний двигун для LOD і org-рішень

Один невеликий цикл, що постійно працює в реальному часі, але **не** є гейплейним тіком — не рухає економіку/бій/лояльність, не переміщує жодну сутність у просторі. Живить кілька незалежних речей одразу:

- **LOD-матеріалізацію/згортання** (§12) — рендер-тіри й sim-fidelity.
- **Cooldown і milestone-логіку** для `BecomePolitical` / `ChangeLeadershipType` / `Expand` / `TakePerk` (Decisions і Perks таби, §5.6) — саме тому ці дії реально працюють у макеті, на відміну від Move/Attack та іншого руху/бою (§7).
- **Реактивний перерахунок** похідних значень у Policy-табі (§5.6) — напр. money income перераховується миттєво при зміні tax rate, без окремого "кроку" симуляції.

Межа проста: якщо дія міняє **число** (capacity, tax rate, чи взято перк) — цей тік її обслуговує. Якщо дія рухає сутність по глобусу або веде до тривалого бою — вона лишається задизейбленою (§7).
